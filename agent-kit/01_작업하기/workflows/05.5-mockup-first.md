# 워크플로우 05.5 — 자립형 목업 우선 시각 게이트 (mockup-first)

> **위치**: `05-reference-intake`(인입·DS시트 승인) **다음**, `03-reference-renewal`(라이브 이식) **이전**.
> **핵심**: 라이브 카페24에 바로 손대기 전에, **카페24와 무관한 자립형 HTML 목업**으로 디자인을 픽셀로 먼저 확정하고 사용자 승인을 받는다.
> **명령**: `/목업` (또는 `/디자인수정` 흐름 내 목업 단계) · 승인 후 → `03-reference-renewal`

---

## 왜 목업을 먼저 만드나 (실증 근거)

라이브 페이지는 카페24 base·모듈 CSS + 과거 커스텀 CSS의 간섭으로 **시안과 100% 일치하지 않는다.** 시안을 바로 라이브에 이식하면 "무엇이 맞는 결과인지" 기준이 없어 whack-a-mole(한 곳 고치면 다른 곳 깨짐)에 빠진다.

**자립형 목업 = "정답 이미지"를 먼저 못박는 장치.** 목업에서 디자인을 픽셀로 확정해 두면, 라이브 이식 단계에서 "이 목업과 1:1로 맞추라"는 명확한 대조 기준이 생기고, 레거시 간섭을 CDP로 걷어낼 때(`06-verify-loop` CDP 전수조사) "무엇을 향해 걷어내는지"가 분명해진다.

---

## 언제 쓰나

- 페이지·컴포넌트 **디자인을 새로 정하거나 크게 바꿀 때** (단순 색·문구 1줄은 `01-quick-fix`).
- 레퍼런스/시안을 라이브에 1:1 이식하기 전 (`05-reference-intake` 승인 완료 후).
- 여러 페이지를 그룹 단위로 통일할 때 (골든 시안 + 병렬 파이프라인).

**스킵 가능**: 이미 확정된 컴포넌트를 형제 페이지에 그대로 적용하는 스모크 단계.

---

## 절차

### 1. 카페24 실제 DOM·데이터 한계 먼저 확인 (목업 이상화 방지)
목업을 그리기 전에 대상 페이지를 개발자도구로 열어 **카페24가 실제로 뱉는 DOM 구조와 데이터 필드**를 확인한다. 시안이 이상화한 구조(예: 여러 거래를 단일 표로 묶기, 존재하지 않는 데이터 컬럼)는 CSS만으로 재현 불가하다. **재현 불가 항목은 목업 설계 단계에서 "구현 상한"으로 표시하고 사용자와 합의**한다(나중에 "왜 시안이랑 다르냐" 방지).

### 2. 자립형(standalone) HTML 목업 제작
카페24와 완전히 분리된 단일 HTML 파일로 만든다.

- `<!doctype html>` + `<meta name="robots" content="noindex,nofollow">`.
- 폰트·아이콘은 **CDN**(Pretendard + Phosphor 등, 그 클라의 하드룰 폰트).
- **inline `:root` 토큰 = 그 클라의 `nk-tokens.css`/`root.css`와 동일한 값**을 그대로 복사(색·간격·radius). 목업과 라이브가 같은 토큰을 써야 이식이 1:1이 된다.
- 클래스는 **짧은 로컬 클래스**(`.gnb`·`.tbl`·`.card`)로 빠르게. 라이브 이식 때 `nk-*` + `.xans-*` 앵커로 번역한다(`03` 3단계). 목업 단계에서 `nk-` 접두사 강제는 불필요.
- 데이터는 **하드코딩 샘플**(실제 `{$변수}` 아님).
- **하드룰 준수**: radius/그림자/기울임 등 그 클라·전역 하드룰을 목업에서부터 지킨다.

**저장 위치**: `clients/{몰}/qa/{작업명}-{YYYYMMDD}/` (예: `.../qa/order-detail-mockup-YYYYMMDD/`).

### 3. PC·모바일 렌더 캡처
목업을 **PC(주력 1440, 보조 1280)와 모바일(주력 390, 보조 375)** 로 렌더해 PNG로 저장(`render/` 또는 `render_pc.png`·`render_mo.png`). 눈으로 검수.

### 4. DS 준수 grep 게이트
목업 HTML에서 하드룰 위반 **0건** 확인:
- `box-shadow` / `text-shadow` / `drop-shadow` 0
- `font-style: italic|oblique` 0
- `translateY(` 리프트/부유 0 (기능 필수 transform 제외)
- 비-포인트색 강조 0 (포인트색은 그 클라 토큰만)

**게이트 통과 = 규칙 준수일 뿐, 완성도 합격은 아니다.** 완성도는 렌더 시각 검수(3)로 판정.

### 5. (선택) 비교 시안 / 갤러리
- 높이·스타일을 여러 안(A/B/C)으로 나란히 만들어 사용자가 고르게 한다.
- 다수 시안은 `index.html`(PC/모바일 나란히 리뷰 갤러리) + 확인 체크리스트로 한 번에 컨펌.

### 6. (대규모) 골든 + 병렬 파이프라인
그룹 단위(여러 아키타입)일 때:
1. 요구사항 확정(강도·범위 질문).
2. **공용 셸 `_shared.css`**(DS 토큰·탭·표·카드·폼·버튼·빈상태·헤더/푸터) + **골든 시안 1장**을 메인이 직접 제작·렌더 검증.
3. 나머지 아키타입을 **병렬 하위 에이전트에 명시 위임**(`_AGENT-BRIEF.md` + 골든 패턴 준수).
4. 메인이 **통합 검수**(grep 게이트 + 전 아키타입 PC/모바일 렌더 + 직접 시각).

### 7. 사용자 승인 게이트 (필수)
- 렌더 + (있으면) 갤러리를 사용자에게 제시 → **「예」 승인 전에는 라이브·FTP·CSS 수정 금지.**
- 승인 후에만 `03-reference-renewal` 3단계(섹션별 라이브 이식)로 진행.
- 「수정」이면 해당 부분만 목업에서 고쳐 재확인.

## 원칙 요약

1. **목업 없이 라이브부터 손대지 않는다** (디자인을 새로 정할 때).
2. **목업 토큰 = 라이브 토큰** — 같은 `nk-tokens` 값으로 1:1 이식 보장.
3. **grep 게이트 통과 ≠ 완성도 합격** — 완성도는 렌더 시각 검수로.
4. **카페24 DOM·데이터 한계를 먼저 확인** — 시안이 이상화한 것은 "구현 상한"으로 합의.
5. **승인 전 라이브 수정 금지.**
