# 모듈 재마크업 방법론 — "오버라이드로 이기지 말고, 구조는 다시 짜라"

> **1급 방법론.** 카페24 스마트디자인에서 base를 CSS로 억지로 덮는 대신,
> **모듈의 동적 기능(데이터 출력·로직)은 그대로 두고 → 원하는 구조로 HTML을 새로 짜고 → 내 CSS를 입히는** 방식.
> 근거: `agent-kit/brain/docs/CAFE24-SMARTDESIGN-AGENT.md` §8-3 승격 + `references/traps.json` + P0 실측(`page-module-map.md`).

**근거 라벨**: `[공식]` 카페24 developers 문서 인용 · `[실측]` kit 실무/라이브 관측 · `[검증필요]` 근거 불충분(추측 금지). 상세 정합은 `references/evidence-labels.md`.

## 0. 한 줄 원칙

> **색·폰트·여백 = CSS 오버라이드 / 레이아웃·구조·배치 = HTML 재작성.** [실측·§8-3]
> base의 복잡한 중첩 마크업을 CSS(flex/grid)로 덮으려 하면 충돌해 깨진다(푸터 flex 강제 → 2단 구조 붕괴 실증). 구조를 바꾸려면 **HTML을 다시 쓴다.**

이유(구조적): 모듈마다 자기 CSS를 `/css` 폴더에서 들고 오고 `[공식]`, 스코프가 없으면 base가 로드순서로 이기며, EZ면 인라인·late-load로 원천 봉쇄된다 `[실측]`. → **오버라이드 일변도는 base와 특이도 전쟁을 무한 반복**한다(traps.json 41종이 그 전쟁의 상처 목록).

## 1. 작업 전 2질문 결정트리

코드를 만지기 전에 **딱 2개만** 판단한다.

```
Q1. 스킨 방식은?   → ez/ 폴더 있으면 EZ(B), 없으면 HTML-native(A)   [skin-method-detect.md]
Q2. 내가 바꿀 건?  → (색·폰트·여백·톤)  vs  (구조·레이아웃·순서·배치)

  색/톤만        → [오버라이드 트랙]  토대정리(nk-cafe24-reset + #nk-skin 스코프 + 토큰)로 base 무력화 → 토큰만 조정
  구조/배치      → [재마크업 트랙]    _nk/inc/ 에 새 마크업 (아래 §2~§4)
  거래 로직 영역  → [오버라이드 트랙 + 반복단위 보존]  (재마크업 대신 안전 우선 — 아래 §5 🔴)
     · 거래 로직 = 옵션/장바구니/주문/결제/회원폼/마이페이지
```

## 2. 핵심 원리 — 무엇을 바꾸고 무엇을 지키나

`module="..."` 블록은 **껍데기(래퍼)이고, 그 안의 HTML 구조는 작성자 소유**다. `[공식]` 편집창 "HTML 보기에서 소스 직접 수정 가능", 반복 블록의 태그·클래스를 작성자가 정의하는 예시 존재.

### ★ 보존 대상 (절대 건드리면 안 되는 것) — `[실측/공식]`
재마크업 시 **다음은 그대로 이식**한다 (§8-3 line 80 기준):
1. 모든 `module="모듈명"` 태그(래퍼) — 지우면 데이터 출력 중단
2. 그 모듈이 **제공하는 `{$변수}`만** 사용 `[공식: 모듈별 변수는 정해져 있음]` — 정의 밖 변수 추가 불가, 모듈 scope 밖에 쓰면 **빈 값**(문자 그대로 아님 — [실측 2026-07-06])
3. **반복 단위**: 소스의 `id="anchorBoxId_{$product_no}"` 류 `<li>` 블록 (§4)
4. 필수 `input name="..."` (폼 제출 키)
5. 폼 액션 `onclick="{$action_*}"` / 구매·결제 바인딩
6. 로그인 분기 `Layout_statelogoff` / `Layout_statelogon`
7. (EZ 스킨인 경우) `data-ez-*` 속성
8. **★반복단위 계층(소속 모듈)** — 반복 `<li id="anchorBoxId_...">`는 **자기 목록 모듈(`product_list*`)의 직계**로 둔다. 반복단위와 목록 모듈 사이에 **다른 `module=` 래퍼를 끼워 넣지 말 것.** 끼우면 반복단위의 "소속 모듈"이 바뀌어 카페24가 상품을 **못 붙인다**(`xans-record-` 0개, `anchorBoxId_` 번호 빈 값). `[실증: 라이브에서 product_normalpackage를 사이에 끼웠다가 상품 바인딩 전면 실패]`

> **⚠️ 자주 하는 실수 — "목록 가격은 변수가 아니다".** 상품목록 카드의 판매가/적립금/요약 등은 `{$product_price}` 같은 변수가 **아니라** `<ul module="product_ListItem">`(관리자 목록표시 설정)가 렌더한다. 없는 변수를 넣으면 빈 값 출력. → 가격/스펙은 **`product_ListItem` 반복 `<li>`를 보존**해서 처리한다. `[실측: 원본 카드에 product_price 0건]`

### 바꿔도 되는 것
- div 중첩·순서·정렬·클래스명(내 nk- 클래스)·배치·반응형 구조 — **구조는 자유** (단 §2-8 계층 규칙 준수: 반복단위는 목록 모듈 직계).

## 3. 재마크업 절차 (5스텝) — `[실측·§8-3]`

```
1. _nk/inc/header.html (또는 footer/섹션)에 "원하는 구조"의 마크업을 새로 작성
     → 골격 스니펫: snippets/_nk-inc/ (반복단위·바인딩 마커 포함)
2. 카페24 필수 요소 이식(§2 보존 대상) — 구조만 바꾸고 데이터 바인딩은 보존
3. layout.html 의 @import(/layout/basic/header.html) → @import(/_nk/inc/header.html) 로 교체
4. _nk/css/ 로 내 클래스 기준 자유 스타일 (base를 이기려 !important 남발 X)
5. 원본은 _nk/_backup_* 로 보존(삭제 금지)
```

> **안티패턴**: base 헤더/푸터의 중첩 마크업을 그대로 두고 CSS flex/grid로 구조를 뒤집기 → 충돌·깨짐. (구조는 CSS가 아니라 HTML로 바꾼다.)

## 4. 반복단위 보존 규칙 (상품목록 등) — `[실측/공식]`

- **`xans-record-`는 스킨 소스에 없다 → 카페24가 렌더타임에 각 레코드에 주입** `[실측: P0에서 소스 234p 중 0건, 라이브 렌더에만 존재]`.
- 소스의 반복 단위는 **`<li id="anchorBoxId_{$product_no}">` 블록**. `[공식: 반복단위가 서로 다르면 "마지막 블록이 count만큼 반복"]`
- → 재마크업 시: **`module="product_list*"` + 반복 `<li>` 블록을 최소 2개(권장) 유지**하고, 그 안 카드 마크업만 새로 짠다. 반복단위를 1개로 줄이거나 없애면 **상품이 1개만 출력**되는 등 깨진다. (traps F13/F4 계열)
- 카드 내부 콘텐츠가 길면 `<!--@import(/_nk/inc/prd.html)-->`로 분리 가능.
- **★라이브 렌더가 최종 게이트.** `check-binding.mjs`(정적검사: module= 있음·anchorBoxId 있음)를 통과해도 **실제 바인딩은 보장되지 않는다**(계층 오류는 정적으로 안 잡힘). 재마크업 후 반드시 라이브에서 **`xans-record-`가 실제로 주입됐는지 + `anchorBoxId_`에 실제 번호가 채워졌는지** 눈으로 확인한다. 확인은 **base가 아닌 임시/신규 스킨**에서. `[실증: 정적검사 PASS했으나 라이브 바인딩 실패]`

## 5. 되는 모듈 / 제약 모듈 원칙 — `[실측 + 2026-07-06 라이브 검증]`

> 등급 기호(`module-safety.json`과 통일): **✅ 완전 재작성 가능** · 🟡 주의 · 🔴 골격 보존.

| 등급 | 모듈군 | 재마크업 지침 | 근거 |
|---|---|---|---|
| ✅ **완전 재작성 OK** | 레이아웃·헤더/푸터(`Layout_*`)·상품 전시(`product_list*`·`product_Imagestyle`·`product_displaycategory`) | `_nk/inc/`로 새 마크업 자유. `module=` 래퍼 + 반복단위(`anchorBoxId` 2개+) + 정의된 변수만 이식 | [검증됨] 카드 그리드·백지 재구축·전면 브랜드 리디자인 성공 |
| ✅ **재마크업 OK (버튼)** | 상세 구매영역·액션 버튼(`product_action`·구매 버튼부) | 새 마크업 + `onclick="{$action_*}"`·`link-*` 속성 **1:1 이식** | [검증됨] 구매버튼 재마크업 → 장바구니 담기·주문 라우팅 성공 |
| 🟡 **주의** | 게시판(`board_*`)·상세정보 표·캘린더 | 반복단위·페이징 변수 유지. 표 구조는 보존 권장(카트와 동형) | [검증필요] |
| 🔴 **표 골격 보존 (de-table 금지)** | 장바구니·주문·합계 상품줄 표(`Order_*`)·회원폼(`member_*`) | **`<table>`·`<tbody module>`·`<tr>`·`rowspan`·`{$form.*}`·`name="{$chk_name}"` 골격 절대 제거·이동 금지.** 바깥만 감싸고 **CSS로만** 스타일. 완전 커스텀 원하면 **JS 재구축**(카페24 카트 숨기고 JS로 데이터 읽어 커스텀 UI + 숨긴 원본에 배선 연결) | [검증됨] 표 de-table 시 주문 깨짐 / JS 재구축은 렌더+주문 성공 |

> **왜 거래 표는 못 뜯나** `[실측 2026-07-06]`: 카페24 스킨엔 `<form>`이 없다 → 엔진이 **`module=` 위치를 보고 런타임에 폼·hidden·`name`을 주입**한다. + `rowspan` 셀병합 + 배송그룹=table 경계. 이 3겹 때문에 표를 뜯으면 주문이 구조적으로 깨진다.
>
> **판정 한 줄**: **표시 모듈 = 반복단위만 지키면 완전 재작성 / 거래 표 = 골격 보존 + CSS**(완전 커스텀은 **JS 재구축** → `cart-js-rebuild.md`).
